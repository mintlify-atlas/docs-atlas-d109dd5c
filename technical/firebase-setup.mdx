## Module Imports

Ripper uses Firebase's modular SDK imported directly from the CDN:

```javascript firebase.js
import { initializeApp } from "https://www.gstatic.com/firebasejs/10.7.1/firebase-app.js";
import { getFirestore, enableIndexedDbPersistence } from "https://www.gstatic.com/firebasejs/10.7.1/firebase-firestore.js";
import { getAuth } from "https://www.gstatic.com/firebasejs/10.7.1/firebase-auth.js";
```

## Offline Persistence

### IndexedDB Persistence

Ripper enables offline data persistence for instant loading on repeat visits:

```javascript firebase.js
enableIndexedDbPersistence(db).catch((err) => {
    if (err.code == 'failed-precondition') {
        console.warn("La persistencia falló: Múltiples pestañas abiertas.");
    } else if (err.code == 'unimplemented') {
        console.warn("El navegador no soporta persistencia de datos.");
    }
});
```

### How It Works

<Steps>
  <Step title="First Visit">
    Data is fetched from Firebase servers and cached in IndexedDB
  </Step>
  <Step title="Subsequent Visits">
    Data loads instantly from local cache while Firebase syncs in background
  </Step>
  <Step title="Offline Access">
    Previously loaded data remains available even without internet
  </Step>
  <Step title="Automatic Sync">
    When connection returns, local changes sync with Firebase automatically
  </Step>
</Steps>

### Error Handling

<Accordion title="failed-precondition Error">
Occurs when multiple tabs are open simultaneously. Only one tab can have persistence enabled at a time. This is a Firebase limitation, not an application error.

**Impact**: Other tabs fall back to memory-only caching.
</Accordion>

<Accordion title="unimplemented Error">
Occurs in browsers that don't support IndexedDB (rare in modern browsers). The app continues to function but without offline capabilities.

**Supported Browsers**: Chrome 24+, Firefox 16+, Safari 10+, Edge 79+
</Accordion>

## Security Considerations

<Warning>
Firebase API keys in client-side code are **not security vulnerabilities**. Security is enforced through:

- Firestore Security Rules
- Firebase Authentication rules
- Domain restrictions in Firebase Console
- Rate limiting and quota management
</Warning>

### Best Practices Implemented

1. **Authentication Required**: All protected routes check auth state
2. **Session Management**: Single-device enforcement (see [Authentication](/technical/authentication))
3. **Data Validation**: Server-side rules validate all database writes
4. **Error Boundaries**: Graceful fallbacks when Firebase services fail

## Troubleshooting

<AccordionGroup>
  <Accordion title="Firebase not loading">
    **Cause**: CDN blocked by network or ad blocker
    
    **Solution**: Check browser console for network errors. Verify `gstatic.com` is not blocked.
  </Accordion>
  
  <Accordion title="Persistence errors in console">
    **Cause**: Multiple tabs open or unsupported browser
    
    **Solution**: Close duplicate tabs or upgrade browser. App continues to work without persistence.
  </Accordion>
  
  <Accordion title="Data not syncing">
    **Cause**: Firestore rules blocking writes or network offline
    
    **Solution**: Check browser network tab and Firebase Console rules. Verify authentication state.
  </Accordion>
</AccordionGroup>
